# Spring AI Platform Agent 代码规范

## 文档目的

本文档用于统一 `agent-service` 及其关联前后端模块的代码风格、职责边界、错误处理、命名方式、测试要求和存量治理方向。

后续与 Agent 相关的优化、重构、评审和问题清理，默认都以本文档作为最低基线执行。

## 适用范围

本文档直接适用于以下内容：

1. `agent-service` 后端代码
2. Agent 工作台、聊天诊断、学习中心相关前端代码
3. Agent 相关 DTO、Mapper、配置对象、测试代码
4. Agent 相关文档、错误文案和运维提示

以下模块虽然不完全受本文档约束，但建议优先对齐：

1. `gateway-service` 中调用 Agent 的适配逻辑
2. `rag-service`、`auth-service` 中与 Agent 联动的共享模型和接口

## 核心原则

### 1. 可读性优先

1. 优先写易读、易排障、易修改的代码。
2. 不为了少写几行代码牺牲语义清晰度。
3. 复杂逻辑应当从方法名、变量名和结构分层上直接体现意图。

### 2. 单一职责优先

1. 一个类只承担一类核心职责。
2. 一个方法只处理一个清晰阶段的逻辑。
3. Controller、Service、Mapper、DTO、Vue 组件不得跨层承载彼此职责。

### 3. 结构化优先

1. 错误码、状态值、风险等级、方向值、筛选值要集中定义。
2. 相同语义不得在多个文件中使用不同写法。
3. 用户文案和底层治理逻辑必须分层维护。

### 4. 稳定性优先

1. 新增治理能力必须考虑并发、超时、失败回收和审计解释。
2. 新增运维入口必须具备可观测、可追踪、可回放能力。
3. 对外返回结构默认只追加字段，不随意破坏现有语义。

### 5. 存量治理优先

1. 优先清理乱码、重复常量、职责混杂和超大文件。
2. 优先修复用户可见和运维关键路径问题。
3. 不允许继续用新增功能掩盖已有规范债务。

## 强制规则

以下规则不满足时，默认不应合入：

1. 禁止提交乱码、不可读字符或编码中间态文件。
2. 禁止使用原始类型 `extends BaseMapper`，必须显式声明泛型。
3. 禁止在 Service 中散落硬编码错误码、风险等级或状态值。
4. 禁止在前端单文件组件中继续堆叠明显可拆分的大段业务逻辑。
5. 禁止把展示文案分散写在多个后端 Service 中。
6. 禁止对可预期业务失败直接抛 `RuntimeException`。
7. 禁止吞掉异常后只返回模糊提示且没有稳定错误码。
8. 关键治理逻辑必须有对应测试覆盖。

## 命名规范

### 1. Java 类型命名

1. 控制器统一使用 `*Controller`。
2. 服务统一使用 `*Service`。
3. 规则判断统一使用 `*Checker`、`*Resolver`、`*Policy`。
4. 数据传输对象统一使用 `*Request`、`*Response`、`*Item`、`*Detail`。
5. 持久化接口统一使用 `*Mapper`。
6. 枚举统一使用 `*Enum` 或明确业务语义名称，不使用含糊缩写。

### 2. 方法命名

1. 查询类方法使用 `get`、`list`、`find`、`build`。
2. 决策类方法使用 `check`、`decide`、`resolve`。
3. 状态变更类方法使用 `create`、`update`、`archive`、`replay`、`recover`。
4. 格式转换类方法使用 `to`、`format`、`normalize`。
5. 禁止使用 `handle`、`process`、`doData` 这类宽泛且无语义的方法名。

### 3. 变量命名

1. 变量名必须体现业务语义。
2. 禁止将 `tmp`、`obj`、`data` 作为关键变量名。
3. 布尔值必须使用可读表达，如 `authorized`、`queueTimedOut`。
4. 集合变量使用复数名，单对象使用单数名。

## Java 后端规范

### 1. Controller 规范

1. Controller 只负责参数接收、基础校验、服务编排和响应封装。
2. Controller 不直接承载复杂业务拼装逻辑。
3. Controller 不直接拼接长段展示文案。
4. Controller 不直接操作 Mapper。

### 2. Service 规范

1. Service 必须保持职责聚焦。
2. 同一个 Service 不应同时承担配置读取、授权判断、状态探测、文案拼装和返回聚合等多类职责。
3. Service 超过 300 行且仍在扩张时，默认需要评估拆分。
4. 单个方法超过 80 行时，默认需要评估拆分。
5. 一个方法中如果同时出现状态维护、等待控制、错误拼装和日志解释，应优先拆为多个私有方法。

### 3. 并发与运行时治理规范

1. 并发控制代码必须明确区分快速通过、进入队列、等待超时和释放资源四个阶段。
2. 可以使用 `synchronized`、`wait`、`notifyAll`，但状态边界必须清晰。
3. 运行时隔离逻辑不得在一个方法里混合状态读写、等待逻辑和响应拼装。
4. 所有并发治理逻辑都必须覆盖成功、拒绝、超时和中断路径测试。

### 4. DTO 规范

1. DTO 只承载数据，不承载业务规则。
2. DTO 字段名必须稳定，避免频繁改名。
3. 对外结构新增字段时优先追加，不直接替换既有字段语义。
4. `reasonCode`、`reasonMessage`、`detail`、`traceId` 各自只承担单一职责。

### 5. Mapper 与 MyBatis 规范

1. 所有 Mapper 必须显式声明泛型参数。
2. 推荐统一写法为 `extends BaseMapper<EntityType>`。
3. 自定义 SQL 方法名必须体现筛选条件和时间范围。
4. Mapper 只定义数据访问能力，不承载业务解释逻辑。
5. 复杂 SQL 优先放入 XML 或显式注解块，不在 Service 中拼接字符串 SQL。

### 6. 常量与枚举规范

1. 错误码、风险等级、方向值、状态值优先沉淀为常量或枚举。
2. 禁止在多个类中重复硬编码 `high`、`medium`、`low`、`ready`、`issue`、`disabled`。
3. 对外暴露的错误码和状态值一旦发布，不随意重命名。

## Vue 前端规范

### 1. 组件拆分

1. 单个页面文件超过 600 行时，默认评估拆分。
2. 单个组件同时负责数据获取、视图渲染、复杂计算和弹窗状态时，默认拆分。
3. 复杂交互优先下沉到组合式函数或独立子组件。

### 2. 状态管理

1. 页面局部状态放组件内部。
2. 跨页面共享状态放 Store。
3. 禁止在多个页面复制同一份业务转换逻辑。

### 3. 文案与接口

1. 用户可见文案集中管理，避免散落在多个组件和服务文件。
2. 接口字段映射必须显式，不依赖隐式结构猜测。
3. 前端错误提示必须区分用户提示和调试信息。

## 测试要求

1. 新增核心治理逻辑必须补单元测试。
2. 修复线上缺陷时必须补回归测试。
3. 并发、降级、路由、审计和权限判断属于高优先级测试区域。
4. 前端复杂转换逻辑和关键交互必须至少覆盖一层测试。

## 当前优先整改项

1. 清理乱码和编码污染文件。
2. 清理控制器中的裸线程池和缺失生命周期管理问题。
3. 收敛散落的状态值、错误文案和风险等级常量。
4. 逐步拆分超大文件：
   `AgentController.java`
   `AgentWorkbenchService.java`
   `MonitorQueryService.java`
   `ChatView.vue`
   `LearningCenterView.vue`
   `AgentWorkbenchView.vue`
5. 为关键治理逻辑补测试，优先覆盖降级、并发隔离、回退和统计聚合。

## 落地顺序

1. 先修可读性和稳定性问题：乱码、线程池、重复状态值、异常处理。
2. 再做结构治理：拆方法、拆类、拆组件。
3. 最后统一测试和文档，把规范沉淀为长期基线。
