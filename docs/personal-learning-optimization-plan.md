# Spring AI Platform 个人学习版优化清单

## 目标

这份文档只保留最适合“个人学习型 AI 项目”的优化项，不追求企业级大而全，重点放在：

- AI 主链路是否真实可用
- 请求过程是否可解释
- 检索效果是否可复盘
- 项目是否方便继续学习和迭代

当前项目已经具备较完整的前后端骨架，下一阶段更值得做的是把“已有能力讲清楚、跑稳定、看明白”。

## 当前判断

这个项目现在已经不缺基础模块，已经有：

- 登录鉴权
- 多助手聊天
- RAG 知识库与文档管理
- 模型网关
- 监控与告警基础能力
- MCP 管理
- 前端控制台和基础测试

真正还值得继续完善的，是下面三类能力：

1. AI 请求闭环是否真正打通
2. 模型 / 工具 / RAG 过程是否可解释
3. 项目是否便于后续继续学习和维护

## 优先级

### P0：最值得优先完成

1. 聊天会话参数真正生效
   - 每个会话支持配置：
     - 模型
     - temperature
     - topK
     - 是否启用 RAG
     - 系统提示词
   - 不只是前后端存储，还要真实影响后端运行时调用

2. 消息级操作补全
   - 重新生成
   - 继续生成
   - 基于某条消息创建分支会话
   - 引用上一条回答继续追问

3. RAG 结构化文件解析稳定化
   - 优先补齐：
     - `csv`
     - `json`
     - `xml`
   - 保证不是“上传成功但无法真正参与检索”

4. RAG 检索解释视图
   - 展示：
     - 用户问题
     - 召回片段
     - 相似度分数
     - 最终答案
   - 让项目具备“为什么答成这样”的学习价值

5. Trace 阶段耗时归因
   - 把一次请求拆成：
     - 鉴权
     - 检索
     - 模型生成
     - 工具调用
     - 存储或日志
   - 不只是知道慢，而是知道慢在哪

6. 工具调用可视化
   - 清楚展示：
     - 工具名称
     - 输入摘要
     - 输出摘要
     - 耗时
     - 成功或失败

### P1：提升完成度和展示效果

7. 模型路由决策解释
   - 本次为什么选这个模型
   - 是否命中降级
   - 候选模型有哪些
   - 是否触发健康状态或权重变化

8. 成本统计
   - 至少按以下维度统计：
     - 模型
     - 用户
     - 助手
   - 展示 token 与估算成本

9. RAG 低质量样本复盘
   - 集中展示低评分问答
   - 标识负反馈证据
   - 便于持续优化提示词、分块策略、召回逻辑

10. 用户使用概览
   - 最近登录时间
   - 最近使用的助手
   - 最近 token 消耗
   - 最近异常记录

11. 全局通知中心
   - 上传结果提示
   - 服务异常提示
   - 后台处理完成提示
   - 告警消息统一收口

12. 快捷键帮助面板
   - 显示当前可用快捷键
   - 包括页面跳转、搜索、聊天输入、命令面板等

### P2：让项目更像一个完整作品

13. 端到端 smoke test
   - 最少覆盖链路：
     - 登录
     - 进入聊天页
     - 上传文档
     - 触发一次 RAG 查询
     - 打开监控页查看结果

14. OpenAPI 文档
   - 统一主要接口说明
   - 便于后续联调、回顾和继续扩展

15. 项目文档体系
   - 至少补三份：
     - 本地启动说明
     - 常见故障排查
     - 系统结构与调用链说明

## 推荐执行顺序

建议按下面顺序推进：

1. 聊天运行时参数闭环
2. RAG 结构化解析与检索解释
3. Trace 归因与工具调用可视化
4. 模型路由解释与成本统计
5. 端到端 smoke test
6. OpenAPI 与项目文档

## 简化原则

作为个人学习项目，下面这些方向不建议过度展开：

- 不必过早追求复杂企业权限体系
- 不必一次性做非常重的流程审批
- 不必为了“像企业项目”而堆很多低频后台能力

更推荐聚焦：

- 一次请求是怎么跑通的
- 一次回答是怎么生成的
- 一次检索为什么命中这些证据
- 一次异常到底卡在什么阶段

## 一句话结论

这个项目下一阶段最值得做的，不是继续加模块，而是把：

- 聊天
- RAG
- 工具调用
- 模型路由
- 监控链路

这 5 条主线做得更可解释、更可验证、更方便复盘。
