# 助手定义与角色关联优化设计

## 1. 目标

本次优化将普通助手的管理模型收敛为与“用户-角色”一致的思路：

1. `ai_agent_definitions` 作为助手主表
2. `ai_roles` 继续作为角色表
3. `ai_agent_roles` 作为助手与角色的关联表

普通助手不再依赖独立的 `ai_bot_permissions` / `ai_bot_permission_roles` 作为主授权路径。  
角色是否可以使用某个助手，直接由 `ai_agent_roles` 决定。  
助手默认 Token 配额、开关、系统提示词等属性直接配置在 `ai_agent_definitions`。

## 2. 最终模型

### 2.1 助手主表 `ai_agent_definitions`

职责：

- 定义助手编码、名称、展示信息
- 保存系统提示词与默认模型
- 保存默认每日 Token 配额
- 保存启停状态与系统保护标记

关键字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 助手主键 ID，新增时由雪花算法生成 |
| `agent_code` | 助手编码，直接作为运行时 `agentType` |
| `agent_name` | 助手名称 |
| `description` | 列表描述 |
| `icon` / `color` | 前端展示信息 |
| `system_prompt` | 系统提示词 |
| `default_model` | 默认模型 |
| `enabled` | 是否启用 |
| `sort_order` | 展示排序 |
| `daily_token_limit` | 助手默认每日 Token 配额 |
| `assistant_profile` | 运行时能力档案，普通助手默认 `generic` |
| `system_defined` | 是否为受保护的特殊助手 |

### 2.2 助手角色关联表 `ai_agent_roles`

职责：

- 记录某个助手允许哪些角色访问

关系：

- 一条助手记录可关联多个角色
- 一个角色可复用于多个助手

这与 `ai_user_roles` 的建模保持对称：

- 用户拥有角色：`ai_user_roles`
- 助手允许角色：`ai_agent_roles`

## 3. 运行时逻辑

### 3.1 权限校验

运行时权限校验改为：

1. 根据 `agent_code` 查询 `ai_agent_definitions`
2. 查询 `ai_agent_roles` 关联到的角色
3. 判断当前用户角色集合是否与助手允许角色集合有交集
4. 若助手被禁用，则直接拒绝访问

不再以 `ai_bot_permissions.allowed_roles` 作为普通助手权限判断主来源。

### 3.2 助手列表

登录后“我可使用的助手列表”改为从以下数据推导：

1. 读取所有启用的 `ai_agent_definitions`
2. 结合 `ai_agent_roles` 过滤当前用户可访问的助手
3. 返回助手编码、允许角色、默认配额等必要字段

## 4. 管理端交互

### 4.1 助手管理

在“助手”管理页直接完成：

- 新建助手
- 编辑助手
- 删除普通助手
- 选择允许角色
- 配置默认每日 Token 配额

不再要求管理员先建助手、再去另一张“权限规则表”里绑定角色。

### 4.2 角色使用统计

角色使用详情调整为两类引用：

- 用户引用
- 助手引用

删除角色前，需要保证：

- 没有用户绑定该角色
- 没有助手绑定该角色

## 5. 特殊助手边界

本次仍保留少量特殊入口：

- `mcp`：特殊助手，保留受保护标记
- `multi`：多智能体运行时入口，不进入普通助手定义表

除特殊入口外，其余普通助手统一按同一模型管理。

## 6. 兼容策略

本次优化后的主路径为：

- 助手定义：`ai_agent_definitions`
- 助手角色：`ai_agent_roles`

历史表：

- `ai_bot_permissions`
- `ai_bot_permission_roles`

暂时保留为兼容遗留代码和存量数据，不再作为普通助手权限配置主入口。后续可按迁移窗口逐步下线。

## 7. 实施结果

本次一次性落地以下内容：

1. 助手定义支持直接维护允许角色与默认 Token 配额
2. 运行时权限校验改读助手角色关联
3. 登录后的可用助手列表改读助手定义与角色关联
4. 管理页移除独立“助手权限规则”主操作路径
5. 角色引用统计改为体现助手引用
